И концепция приложения меняется. Наверно это должно быть  как отдельное приложение для windows 10, на python. Общий вид окна, дизайн  должен остаться как сейчас, но с возможностью переключатся на светлую тему и обратно.

Структура: 
Главная страница (Список заказов и другая полезная информация)
-> Страница заказа 
->-> Создание/редактирование заказа 
->-> Создание спуска полос

Должна быть возможность переключатся между страницами.

На главном экране есть список заказов (стольбцы: номер, название на русском, дата обновления, дата создания, статус (мониторится папка или нет, красным если да). При нажатии на заказ, открывается страница заказа. На этой странице я создаю новый заказ по этому алгоритму:
1. Я загружаю через drug and drop в приложение файл или pdf или jpg с спецификацией заказа. Пример прилагаю: 
	пример спецификации заказа.pdf
	пример спецификации заказа.jpg
	реже возможны другие варианты спецификаций.
2. Приложение распознаёт текст на спецификации, все от верха документа до строки "Дата в печать" включительно, данные появляются на экране. Их можно редактировать.
3. В окне есть зелёная кнопка - "Создать". После её нажатия:
	Инфа пишется в sqlite3 файл. С возможностью в дальнейшем поменять на удалённый Postgresql.
	Структура такая:
		CREATE TABLE "printery_order" ("id" integer NOT NULL PRIMARY KEY AUTOINCREMENT, "number" integer NOT NULL UNIQUE, "type" varchar(3) NOT NULL, "circulation" integer NULL, "binding" varchar(4) NOT NULL, "width" integer NULL, "height" integer NULL, "created" datetime NOT NULL, "delivery_date" datetime NULL, "submiting_files" datetime NULL, "due_date" datetime NULL, "name" varchar(32) NOT NULL)
		CREATE TABLE "printery_order_owner" ("id" integer NOT NULL PRIMARY KEY AUTOINCREMENT, "order_id" bigint NOT NULL REFERENCES "printery_order" ("id") DEFERRABLE INITIALLY DEFERRED, "user_id" bigint NOT NULL REFERENCES "printery_user" ("id") DEFERRABLE INITIALLY DEFERRED)
		CREATE TABLE "printery_paper" ("id" integer NOT NULL PRIMARY KEY AUTOINCREMENT, "name" varchar(64) NOT NULL, "type" varchar(3) NOT NULL, "width" integer NULL, "height" integer NULL, "density" integer NULL)
		CREATE TABLE "printery_part" ("id" integer NOT NULL PRIMARY KEY AUTOINCREMENT, "part_name" varchar(3) NOT NULL, "pages" integer NULL, "color" varchar(3) NOT NULL, "laminate" varchar(3) NOT NULL, "uflak" bool NOT NULL, "order_id" bigint NOT NULL REFERENCES "printery_order" ("id") DEFERRABLE INITIALLY DEFERRED, "paper_id" bigint NULL REFERENCES "printery_paper" ("id") DEFERRABLE INITIALLY DEFERRED)
		
		number это четырех-значный номер заказа.
		
		Если нет данных, то пропускать. Ещё содавать в папке in xml файл с этими данными. Имя файла как название папки: 0001_ShortName
4. Создаётся папка на сетевом диске "P:\". Имя вида 0000_Короткое_название_заказа_латинецей до 40 символов. 0000 - номер заказа.
	В ней содаётся папка "in". в которой будут лежать файлы заказчика.
	С этого момента папка P:\0001_NameOfOrder\in начинает мониториться приложением. 
	На странице заказа появляется красная кнопка мониторинга. Если на неё нажать мониторинг этой папки выключится.
	После того как в папке in появляются pdf заказчика, 
			они копируются в папку d:\Pitstop_\out\, в папку вида 0001_NameOfOrder, после проверки в Enfocus PitStop Server 22.1.0, в паке d:\Pitstop_\Log\ появляются папка заказа с pdf файлом с логом префлайта и xml файл, если есть ошибки. На экране приложения должны отобразится результаты проверки pitstop из xml файла (обрезной формат страниц, кол-во страниц в файле, красочность и список, если есть, Errors и Warnings. 
			Паралельно в папке P:\0001_NameOfOrder\in запускается рефайн этих файлов в Kodak Prinergy Evo 6.1.3.0. (надо как то настроить, чтобы после рефайна новые pdf появлялись в той же папке in).
			
5. На странице заказа должна быть кнопка "Сделать спуск", при нажатии открывается новая страница. Здесь можно загрузить jpg фото рукописно рисованный технологом спуск полос на бланке. Должна быть возможность повернуть фото, если она не той ориентации. Должна быть возможность увеличения фото для просмотра. Наверно лучше просмотр фото разместить на боковой панели. Пример файла:
	пример спуска1.jpg
	пример спуска2.jpg
Должна быть кнопка "Распознать". При нажатии фото спуска должно быть по возможности распознано с помощью локальной модели ИИ. Информация должна отобразиться на странице с возможностью редактирования:
	кол-во полос в блоке, тип скрепления (термоклей, скрепка, шитьё) если есть, обрезной формат изделия, красочность (например 4+4, 1+1, 4+1). Эти данные нужно сравнить с инфой из спецификации, при не совпадении как предупредить - может подсветить красным), 
	формат бумаги на блок (например 720x1040мм или 65x47см.), красочность печати. Так же на обложку, вклейки, форзацы (если они есть).

Есть 2 места где хранятся шаблоны preps:
	p:\Preps\Templates\
	\\NAS-PREPRESS\Archives\!!!_Preps_Templates\

Их имен часто например такие 1662_Comix_160x245_72x52_Tkley.tpl, где 1662 - номер заказа, Comix - имя заказа, 160x245 - обрезной формат, 72x52 - основной формат бумаги	или блока, Tkley - тип скрепления (Tkley - термоклей, Skrepka - скрепка, Shitie - шитьё).

Надо просканировать все шаблоны и найти наибольшие соответсвия и совпадения по данным с новым заказом.
Вывести список найденных шаблонов с путём к файлу, с возможностью по клику открывать каждый шаблон в preps (как в проводнике windows).

В папке заказа создать пустой файл job preps вида: название папки заказа в P:\ .job
